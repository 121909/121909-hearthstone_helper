### 2.1 牌表锁定规则

1. 使用 `CardId + 数量` 计算目标牌表哈希，不依赖本地化名称。
2. 仅在 `RANKED_WILD` 且本方完整牌表哈希匹配时启用。
3. 同时记录 Hearthstone build、HDT 版本和 HearthDb/CardDefs 哈希。
4. 任一卡牌文本、标签或费用发生变化时，将规则集标记为 `unsupported_patch`，停止生成回合路线。
5. 卡组不匹配时不做模糊推断，也不以“相似弃牌术”模式继续运行。

## 3. 产品目标与边界

### 3.1 MVP 目标

- 在己方回合给出一条完整主路线，包括出牌、弃牌目标、攻击目标、随从位置、英雄技能和结束回合。
- 最多提供一条备选路线，并说明它相对主路线的关键取舍。
- 对发现、检视、定向弃牌等交互，在客户端实际给出候选后生成下一阶段建议。
- 标注建议置信度、随机结果范围、关键假设和自伤风险。
- 局面变化后立即使旧建议失效，避免显示过时行动。
- 网络或大模型不可用时，仍能依靠本地规则引擎返回合法路线。

### 3.2 非目标

- 不自动控制鼠标、键盘或游戏客户端。
- 不宣称在隐藏信息和随机性存在时得到数学意义上的全局最优解。
- MVP 不支持其他牌组、对手视角、观战或酒馆战棋。
- 不使用玩家正常对局中不可见的 CardId、对手手牌或牌库顺序。
- 不让大模型直接生成未经规则引擎验证的动作。

### 3.3 “最优”的工程定义

MVP 中的“最优路线”指：在当前可见信息、指定搜索预算、当前规则集和对手信念模型下，综合评分最高的合法行动序列。正常天梯对局只能求期望最优；只有完全确定且规则覆盖完整的战术题才可能证明搜索范围内最优。

## 5. 状态采集设计

### 5.1 接入方式

优先编写 HDT 插件，不重新解析完整 `Power.log`。插件通过 `Hearthstone_Deck_Tracker.API.Core.Game` 读取 HDT 已维护的 `GameV2`、`Player`、`Entity` 和 `GameTag`，并订阅：

- `OnGameStart` / `OnGameEnd`
- `OnTurnStart`
- `OnPlayerDraw` / `OnPlayerPlay`
- `OnOpponentPlay`
- 选择题与模式变化相关事件

`IPlugin.OnUpdate()` 只做轻量状态检查，不执行网络调用或搜索。

### 5.2 快照触发与稳定性

1. 关键游戏事件到达后标记状态为 dirty。
2. 状态连续 150-250ms 未变化后生成快照，避免在一个结算链中间求解。
3. 快照构建时复制实体和标签，禁止 Worker 直接持有正在变化的 `GameV2` 引用。
4. 对快照进行规范化排序并计算 `state_id`。
5. 相同 `state_id` 不重复求解；新状态产生时取消旧请求。
6. Overlay 只接受与当前 `state_id` 完全一致的结果。

### 5.3 Snapshot 最小字段

- 协议版本、规则版本、Hearthstone build、卡牌定义哈希。
- 对局 ID、回合、步骤、当前行动方、剩余回合时间。
- 双方英雄的生命、护甲、攻击、冻结状态、武器和英雄技能。
- 本方当前/临时/已用法力和最大法力。
- 本方手牌实体：实体 ID、CardId、动态费用、区域位置、创建来源、临时状态和全部必要标签。
- 双方场面实体：位置、攻击、剩余生命、关键词、攻击次数、冻结/休眠/嘲讽/突袭状态和附魔。
- 地点耐久与可用状态。
- 本方原始牌表、已知剩余牌库、墓地、已弃卡牌及累计弃牌次数。
- 对手手牌数量、已公开卡牌、奥秘候选集合及可见的任务/地点。
- 本回合已执行动作和当前客户端提供的 Choice 候选。
- 栉龙抽取实体绑定、临时牌截止回合、续连熵能产生的时空撕裂数量等专用派生状态。

### 5.4 可见性与隐私

- 采用字段白名单，不序列化完整 `Entity` 对象。
- 对手未知手牌只记录数量和概率分布，不记录日志中可能泄露但 UI 不可见的 CardId。
- 不发送 BattleTag、AccountId、ServerInfo、GameHandle、聊天或本地文件路径。
- 本地诊断日志默认脱敏；上传训练或评估数据必须显式选择加入。

## 6. 规则引擎

### 6.1 动作模型

MVP 至少支持：

```text
PlayCard(sourceEntityId, targetEntityId?, boardPosition?)
Attack(sourceEntityId, targetEntityId)
UseHeroPower(targetEntityId?)
UseLocation(sourceEntityId, selectedEntityId)
SelectChoice(choiceId, selectedEntityId)
EndTurn
```

每个动作必须经过前置条件验证，并产生新的不可变状态、事件列表、随机分支和可解释的评分增量。

### 6.2 最小通用规则

- 法力、动态费用、手牌上限和七个场面格。
- 英雄、随从、武器、地点和英雄技能的基本可用性。
- 攻击次数、冻结、嘲讽、突袭、随从刚入场限制和目标合法性。
- 抽牌、爆牌、疲劳、召唤失败和死亡后的场面空间释放。
- 卡牌文本顺序、战吼、亡语、弃牌触发和抽到即施放的结算顺序。
- 随机分支可复现：测试使用固定 seed，生产使用多 seed 采样。

### 6.3 该牌组的强制规则覆盖

#### 弃牌来源

- 魔眼秘术师：玩家选择一张手牌弃掉。
- 维希度斯的窟穴：从实际展示的三张牌中选择一张弃掉，再抽两张。
- 邪恶低语：弃掉最低费用牌；最低费并列时建立随机分支。
- 时空之爪：英雄攻击结算后弃掉最高费用牌；最高费并列时建立随机分支。
- 灵魂之火：造成伤害后随机弃掉一张手牌。
- 栉龙：跟踪其战吼实际抽到的实体，并在该栉龙亡语时弃掉该实体。
- 回合结束：正确结算灵界和咒怨之墓产生的临时牌。

#### 弃牌收益

- 古尔丹之手：使用或弃掉均抽三张，计算爆牌和疲劳风险。
- 助祭耗材：使用或弃掉均召唤两个随机一费随从；场面不足时按真实召唤顺序处理。
- 灵魂弹幕：使用或弃掉均产生五点随机分配伤害，逐次更新可选敌方目标池。
- 骨网之卵：弃掉时触发亡语并尝试召唤两个 2/1。
- 镀银魔像、行尸：弃掉时尝试召唤，受场面空间限制。
- 地狱公爵：累计每次有效弃牌带来的 +2/+2，并处理突袭目标。

#### 顺序与资源

- 邪恶低语必须验证“先弃牌、后全场 +1/+1”，包括弃牌后新召唤的单位是否获得增益。
- 魔眼秘术师先占用一个场面格，再结算定向弃牌及相关召唤。
- 派对邪犬占用三个场面格并造成三点自伤。
- 续连熵能先增益当前场面，再洗入两张时空撕裂。
- 时空撕裂抽到即对己方英雄造成三点伤害，并影响后续抽牌与生存评分。
- 灵界和咒怨之墓的临时牌必须在本回合内完成“使用、转化为有益弃牌或接受损失”的比较。
- 灵魂之火先处理四点目标伤害，再处理随机弃牌；斩杀终止条件必须与真实事件顺序一致。

### 6.4 未覆盖交互的处理

遇到未知卡、未知标签或未实现的随机一费随从效果时：

1. 不猜测合法动作。
2. 将对应分支标为 `unsupported_interaction`。
3. 降低整条路线置信度，必要时不显示建议。
4. 保存脱敏 fixture，作为下一次规则扩展输入。

## 7. 候选生成与搜索

### 7.1 候选动作生成

- 枚举所有当前合法出牌、目标和随从位置。
- 枚举所有合法攻击和英雄攻击目标。
- 对魔眼秘术师、窟穴和 Choice 使用客户端实际提供的候选。
- 对随机弃牌只建立结果分支，不让搜索器选择随机结果。
- 生成牌组专属宏动作，优先探索：
  - 弃掉收益牌后铺场；
  - 临时牌转化；
  - 弹幕/灵魂之火斩杀；
  - 先召唤再群体增益；
  - 先腾场面格再弃牌召唤；
  - 古尔丹之手过牌链；
  - 地狱公爵成长后的突袭交换。

### 7.2 搜索策略

首版采用 Beam Search，便于控制延迟并调试：

- `beamWidth` 初始值 64。
- 单回合最多搜索 12 个原子动作。
- 本地首轮时间预算 200-300ms。
- 使用规范化状态哈希做 transposition 去重。
- 剪除在法力、场面、手牌、伤害和资源上被严格支配的路线。
- 确定性斩杀搜索优先于一般价值搜索。
- 随机弃牌、弹幕和随机一费随从使用精确小分支或 Monte Carlo。
- 同时保留期望值、最差分位数和方差，避免高风险路线仅凭均值胜出。

### 7.3 对手信念模型

MVP 不需要重建所有狂野牌组，先提供三个风险档：快攻、资源/控制、组合技。根据职业、已公开卡牌和回合节奏更新概率，用于调整：

- 自伤容忍度；
- 是否优先嘲讽和交换；
- 是否保留直伤；
- 是否使用英雄技能；
- 对清场风险的铺场惩罚。

后续再引入按补丁版本维护的狂野 archetype profile。

## 8. 评分模型

评分应保留独立维度，避免一个总分掩盖高风险：

1. 本回合确定性斩杀。
2. 本回合概率斩杀及失败后的局面。
3. 对手下回合斩杀风险和己方有效生命。
4. 场面攻击、生命、嘲讽、可立即攻击单位和站位质量。
5. 弃牌触发收益减去误弃核心牌成本。
6. 手牌数量、可用资源、过牌、爆牌和疲劳风险。
7. 临时牌未利用损失。
8. 场面空间、召唤浪费和增益覆盖数量。
9. 灵魂之火/灵魂弹幕等直伤储备。
10. 派对邪犬、英雄技能和时空撕裂造成的自伤风险。
11. 地狱公爵后续成长价值。
12. 对手类型对应的清场、抢血和组合技时间压力。

初始权重由规则和专家标注设定。积累足够授权对局后，再用 pairwise ranking 学习权重；不直接用最终胜负反向归因单个回合。

## 9. 大模型集成

### 9.1 职责限制

大模型只接收规则引擎生成的 Top-K 合法路线及压缩局面摘要，用于：

- 对接近的候选进行策略层重排；
- 结合对手信念解释节奏与风险；
- 生成简短中文行动说明。

大模型不得创建新的实体 ID、动作、目标或随机结果。

### 9.2 请求与响应协议

示例响应：

```json
{
  "state_id": "turn-8:7f91",
  "selected_candidate_id": "route-3",
  "alternative_candidate_id": "route-1",
  "confidence": 0.76,
  "summary": "先定向弃掉古尔丹之手补充资源，再利用新抽到的低费牌铺场。",
  "risk": "若连续抽到时空撕裂，本回合后有效生命偏低。"
}
```

客户端必须验证：

- `state_id` 仍然是当前局面；
- candidate ID 属于本次请求；
- 候选路线再次通过合法性校验；
- 响应未超时且协议版本一致。

### 9.3 降级策略

- 本地搜索结果应先于大模型解释显示。
- LLM 超时、限流或离线时保留本地建议。
- 相同状态和候选集合可缓存解释。
- 建议 LLM p95 响应目标小于 2.5 秒；超过局面有效期的响应直接丢弃。

## 10. Overlay 交互计划

Overlay 是高频工具界面，应保持安静、紧凑并避免遮挡棋盘：

- 默认只显示主路线的 3-5 个步骤；更长路线可展开。
- 每一步显示动作图标、卡牌/实体名称、目标和次序，不依靠颜色单独表达含义。
- 备选路线折叠显示，只突出关键差异。
- 固定面板尺寸和稳定行高，异步结果不得造成布局跳动。
- 状态包括：分析中、建议可用、状态已变化、离线、本补丁不支持、交互未覆盖。
- 玩家执行任一动作后，旧路线立即降低显示优先级并触发重算。
- 低动画，内容替换仅使用 150-200ms 淡入淡出，并支持减少动态效果。
- 文本与背景对比度至少满足 WCAG AA，键盘焦点可见。
- 错误、风险和置信度除颜色外还必须有文字或图标提示。
- Overlay 只显示建议，不提供自动执行入口。

## 12. 测试计划

### 12.1 单卡测试

17 个单卡条目均测试：

- 正常使用；
- 无合法目标或无足够场面空间；
- 动态费用；
- 手满、场满和牌库空；
- 使用与弃掉两种路径；
- 临时牌和卡牌被变形/复制时的实体跟踪。

### 12.2 关键组合测试

- 邪恶低语弃骨网之卵后，新蜘蛛是否得到增益。
- 魔眼秘术师占位后弃行尸/镀银魔像时的场面空间。
- 时空之爪最高费并列时的概率分支。
- 灵魂之火随机弃古尔丹之手或灵魂弹幕后的完整触发链。
- 灵界抽到多个弃牌收益牌时的最佳次序。
- 咒怨之墓产生临时古尔丹之手/灵魂弹幕时的使用与弃掉比较。
- 古尔丹之手连续过牌导致爆牌或抽到时空撕裂。
- 栉龙死亡时，绑定牌已经使用、变形、被再次弃掉或不在手牌。
- 灵魂弹幕导弹击杀随从后目标池变化及概率斩杀。
- 助祭耗材随机召唤导致的场面上限和静态效果。
- 地狱公爵在同一结算链多次弃牌后的身材与突袭交换。
- 派对邪犬、自残英雄技能和时空撕裂共同造成的自杀路线剪枝。

### 12.3 属性与不变量测试

- 手牌永不超过 10，场面单位永不超过 7。
- 法力不可为负。
- 同一实体不能重复使用或攻击超过合法次数。
- 搜索产生的每个动作都能由合法性校验器重新接受。
- 固定 Snapshot、规则版本和 seed 必须得到相同结果。
- 未知交互不得生成伪造动作。

### 12.4 集成与回放测试

- `Gameplay.Start`、重连、回合切换、Choice、结束对局。
- 单个结算链中间不应生成可见建议。
- 动作后旧 `state_id` 结果必须被拒绝。
- LLM 超时、错误 JSON、无效 candidate、限流和断网。
- HDT 关闭、插件卸载和下一局开始时取消所有 Worker。

## 13. 验收标准

- 目标牌表识别准确率 100%，非目标牌表不启动。
- 规则覆盖清单中的 17 张牌全部通过测试。
- 在回放 fixture 和影子运行中，输出动作合法率 100%。
- 过期建议展示次数为 0。
- 本地首轮建议延迟 `p95 < 300ms`。
- 插件更新线程不执行同步网络或长时间搜索。
- 未覆盖交互必须明确降级，不能编造结果。
- 至少 200 个专家标注局面，Top-3 路线一致率目标不低于 80%。
- 至少 50 局影子运行无状态错位、重复请求或明显 HDT 卡顿。
- 发送给模型的请求不含账号、服务器和本地路径等个人数据。

## 14. 主要风险与缓解措施

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 游戏补丁改变卡牌 | 规则结果错误 | build 与 CardDefs 哈希门禁，变化即停用 |
| HDT 插件 API 变化 | 插件无法加载或状态缺失 | 固定兼容版本，建立启动自检和适配层 |
| 日志结算期间状态不完整 | 建议基于中间状态 | dirty/debounce、不可变快照、Choice 分阶段求解 |
| 随机一费随从池复杂 | 期望值偏差 | 按补丁生成卡池、采样、未知效果降低置信度 |
| 对手隐藏信息 | 路线并非全局最优 | 信念分布、风险分位数、明确显示假设 |
| LLM 延迟或幻觉 | 建议迟到或非法 | 本地结果优先、只允许选择 candidate、再次校验 |
| 高搜索分支数 | 超过回合内延迟预算 | 宏动作、Beam Search、支配剪枝、取消旧任务 |
| Overlay 遮挡或跳动 | 影响正常操作 | 固定尺寸、高密度布局、低动画、可快速隐藏 |
| 合规风险 | 功能无法发布 | 仅建议、不自动操作，阶段 0 完成规则审查 |

## 15. 数据与持续改进

### 15.1 初始数据

HSReplay 页面提供牌表和聚合卡牌统计，但不提供逐回合专家行动轨迹。MVP 需要另外准备：

- 该牌组的个人或获得授权的 `.hdtreplay`；
- 脱敏的关键局面 Snapshot；
- 专家对 Top-3 路线的排序和理由；
- 每个补丁版本的对手类型和高频清场风险配置。

### 15.2 反馈记录

在用户明确选择加入时记录：

- Snapshot 哈希和规则版本；
- Top-K 候选评分；
- 实际执行动作；
- 建议是否过期；
- 对局最终结果；
- 用户对建议的接受/拒绝标记。

不应把最终胜负简单归因到单个回合。模型训练优先使用同一局面下的 pairwise 路线偏好和专家标注。

## 16. 发布策略

1. `offline`：仅回放和 fixture，不连接实时游戏。
2. `shadow`：实时计算但不显示建议，只收集本地诊断。
3. `experimental`：对少量用户显示建议，默认本地搜索，LLM 可关闭。
4. `beta`：达到验收指标后开放，保留补丁门禁和一键停用。
5. 后续版本再评估支持相近弃牌术、留牌建议和对手 archetype 数据。

## 17. 首个可执行迭代

第一迭代建议控制在 5 个工作日内：

1. 创建工程骨架和目标牌表 profile。
2. 完成精确牌表识别、HDT 事件订阅和 Snapshot 导出。
3. 只实现“当前局面摘要 + 合法攻击/结束回合”，验证状态时序。
4. 为魔眼秘术师、邪恶低语、时空之爪和灵魂之火实现四类弃牌选择。
5. 建立固定 seed 的规则测试和三个脱敏黄金局面。

迭代结束条件：插件能在目标牌组的己方回合稳定生成版本化 Snapshot，动作后旧结果不会显示，且四类弃牌来源的候选与实际游戏一致。
